# 台灣五都垃圾車開放資料整合指南

## 目的 (Purpose)
本指南旨在提供開發者一套標準化的流程與邏輯說明，用以整合台灣主要五大城市（台北、新北、台中、台南、高雄）的垃圾清運開放資料。

## 作用 (Role)
本文件扮演技術備忘錄與實作準則的角色，詳細說明了：
1. **API 位址**：列出各城市的 API 串接端點。
2. **差異化處理**：針對各城市資料格式與欄位的特殊性（如 CSV/JSON、大小寫差異）提供實作建議。
3. **系統保護**：提供本地快取與頻率限制等最佳實作，以確保系統的穩定性。

## 2. 南台灣 (台南、高雄) 的特異性處理

### A. 台南市：欄位不對稱
*   **大小寫差異**: 靜態班表為全大寫 (`LATITUDE`), 即時動態為全小寫 (`y`)。
*   **排序依據**: 必須使用 `ROUTEORDER` 進行站點連線，不可僅依 API 回傳順序。

### B. 高雄市：高不穩定性
*   **合併座標**: 處理 `"22.61,120.30"` 這種合併字串，需手動拆分。
*   **時間格式**: 處理 `"0830"` (補冒號) 或 `"16:00-16:10"` (取起始時間) 的混雜情況。

---

## 3. 北中台灣 (台北、新北、台中) 的特異性處理

### A. 台北市：巨量 JSON 與複合識別
1.  **參數限制**: 請求時必須包含 `limit=20000`。若未指定，預設回傳筆數（通常為 20-100 筆）完全不足以覆蓋全台北市站點。
2.  **唯一識別 (Key)**: 台北市同一條路線（如「木柵線」）可能同時有 3 台車在跑。
    *   **邏輯**: 應使用 `路線名稱 + 車號` 作為 Model 的 Key，否則地圖上的車輛會因重複 ID 而不斷跳動或被覆蓋。
3.  **時間格式**: 常出現 `"2030"` (無冒號) 格式，需統一轉換為 `"20:30"` 以利解析。

### B. 新北市：安全門檻與 CSV 性能
1.  **安全性驗證 (403 Forbidden)**: 新北伺服器會嚴格檢查 `User-Agent`。若使用預設的 Http Client（如 Dart 的 `http` 或 Python 的 `requests`）且無自定義 Header，請求會被拒絕。
    *   **必要 Header**: `{'User-Agent': 'Mozilla/5.0...', 'Referer': 'https://data.ntpc.gov.tw/'}`。
2.  **解析效能**: 新北提供的是 CSV 格式且資料量巨大（超過 10MB）。
    *   **邏輯**: 嚴禁在主線程 (Main Thread) 解析 CSV。必須使用 **背景 Worker** 或分段讀取 (Stream)，否則 App 會出現明顯的卡頓甚至系統崩潰 (OOM)。
3.  **髒資料過濾**: 新北 CSV 中常有座標為 `0` 或空字串的廢棄站點，解析時需實作 `tryParse` 並丟棄非法座標。

### C. 台中市：時間戳記與車牌映射
1.  **ISO T 格式**: 時間欄位常帶有日期，如 `"20240411T093000"`。
    *   **處理**: 應取 `T` 之後的 4 位數，並格式化為 `09:30`。
2.  **跨 API 映射**: 台中市將「車牌號碼」作為動態與靜態資料的唯一連結紐帶。
    *   **策略**: 預載班表時以車牌為索引建立 Map，接收動態 API 時再將經緯度更新至對應的車牌 Model。

---

## 4. 系統保護與優化建議

1.  **本地快取 (Caching)**: 台北與新北的靜態班表資料量大，同步完成後應存入 SQLite 並設置版本號。
2.  **存取節流 (Throttling)**: 即時 API 更新頻率應維持在 30-60 秒，並僅在用戶正在查看地圖時啟動。
3.  **錯誤回退 (Fallback)**: 當 API 斷訊時，利用本地資料庫的表定時間推算「車輛理論位置」，並在地圖上以半透明圖示顯示，避免用戶誤以為系統掛掉。
4.  **編碼一致性**: 務必使用 `utf8.decode(response.bodyBytes)`。

## 5. 開發筆記
*   **事務處理**: 存入新北/台北資料時，使用 `Transaction` 寫入 SQLite。
*   **範圍驗證**: 過濾任何緯度不在 22-26、經度不在 120-122 之間的錯誤座標。
